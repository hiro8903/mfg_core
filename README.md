# MfgCore - 製造・受託加工業向け 統合基幹システム

お客様からお預かりした基材の受託加工フローを中心に、人事組織・品質管理・在庫・調達プロセス等、製造業の基幹業務を一元管理する統合基幹システムです。

> **⚠️ Disclaimer (免責事項)**
> 本リポジトリは、プログラミング初学者による**ポートフォリオ（学習用プロジェクト）**です。
> 架空の製造業・受託加工業を想定し、「もし自分が現場の基幹システムをゼロから設計・開発したらどうなるか」というテーマで、モダンなRailsの技術（Hotwire, Tailwind CSS v4等）とドメインモデリング（DDD）の学習を兼ねて作成しています。
> 実業務での使用を想定したものではなく、コードや設計に至らない点が含まれる可能性がありますが、アーキテクチャ設計（ADR）やER図など、ドキュメント主導の開発プロセスを実践する練習の場として公開しているものです。
>
> **🤖 AI ツールの活用について**
> 本プロジェクトの設計・実装・ドキュメント作成には、AI搭載コーディングエディタ（Google Antigravity）および複数のLLMモデルをペアプログラミングのパートナーとして積極的に活用しています。
> コードや設計判断のすべてが自力のみによる成果ではなく、AIとの対話を通じて学びながら構築したものです。

## 🗺️ 1. 開発トップポータル（ドキュメント・マップ）

本プロジェクトは **「コードと同等にドキュメント（設計の文脈）を重視する」** 方針で開発されています。開発を進める上で必要な情報は、すべて以下の各ドキュメントに集約されています。

### 🧭 羅針盤・プロジェクト管理
* **[`PLAN.md`](./PLAN.md)**
  * **プロジェクトの地図。** 次のステップと現在地のタスク進捗（TODOリスト）を管理。迷ったらここを見ます。

### 📚 用語と設計図（ドメイン・アーキテクチャ）
* **[`docs/glossary.md`](./docs/glossary.md)**
  * **用語集。** システム内で使うモデル名や業務概念（取引先 vs 納入先など）の正となる定義。
* **[`docs/architecture/er_diagram.md`](./docs/architecture/er_diagram.md)**
  * **目標とする論理データ構造（ER図）。** 要件定義に基づき、将来的に目指すDBスキーマを定義。
* **[`db/schema.dbml`](./db/schema.dbml)**
  * **実績のDB構造。** マイグレーションを実行後、`bin/rake dbml:export` で自動生成される実際のスナップショット。
* **[`docs/architecture/adr/`](./docs/architecture/adr/)**
  * **アーキテクチャ・設計判断記録（ADR）。** 「なぜその設計・技術を選んだのか」という理由とトレードオフの軌跡。（例：認証方式、論理削除、統合マスタモデルなど）

### 🎨 フロントエンド・ルール
* **[`docs/design/design_system.md`](./docs/design/design_system.md)**
  * **UI・デザインシステム。** ボタンやバッジの「色の意味（用途）と Tailwind クラス」を定義したガイドライン。

## ✅ 2. 【最重要】開発完了のチェックリスト (Definition of Done)

コードを書くだけでは開発は終わりません。新しい機能や **DBモデル（テーブル）を追加・変更した際** は、コミット前に必ず以下のプロセス（更新漏れがないか）を確認してください。

- [ ] **DBMLの自動更新**: `rails db:migrate` 実行後、設計図を最新化しましたか？
  - 💻 `bin/rake dbml:export` を実行する。
- [ ] **用語集の更新**: 新しいモデルや業務概念（例：仕入見積）を実装した際、辞書に追記しましたか？
  - 📝 [`docs/glossary.md`](./docs/glossary.md) を更新する。
- [ ] **ADRの作成検討**: なぜ他ではなくその設計（あるいは技術）を選んだのか、未来に残すべき判断がありましたか？
  - 📝 [`docs/architecture/adr/`](./docs/architecture/adr/) に新規作成する。
- [ ] **ロードマップの更新**: 今回の機能実装が完了したことを計画表に記録しましたか？
  - 📝 [`PLAN.md`](./PLAN.md) のチェックボックスに `[x]` を入れる。
- [ ] **UIルールの遵守**: 新しい画面を作る際、定義済みの色ルールや共通パーシャル（`_button`, `_badge`, `_form_errors` 等）を使いましたか？
  - 📝 逸脱（あるいは拡張）が必要だった場合は [`docs/design/design_system.md`](./docs/design/design_system.md) を更新する。

## 🚀 3. ローカル開発環境の起動方法

### 初回セットアップ
```bash
$ bin/setup
$ bin/rails db:seed  # 初期テストデータの投入
```

### サーバー立ち上げ (Hotwire & Tailwind v4)
```bash
$ bin/dev
# ブラウザで http://localhost:3000 にアクセス
```

## 🧪 4. テストとCI/CD

GitHub へプッシュ/PR を作成すると、GitHub Actions（`.github/workflows/ci.yml`）により以下の自動チェックが走ります。
* **RSpec / Minitest**: ユニットテスト・システムテスト
* **RuboCop**: コード規約の検査（スタイル違反がないか）
* **Brakeman / Bundler Audit**: コードと依存関係（Gem等）のセキュリティ脆弱性スキャン

ローカルで事前にチェックする場合は以下を実行してください。
```bash
$ bin/rails test
$ bin/rails test:system
$ bundle exec rubocop
$ bin/brakeman --no-pager
$ bin/bundler-audit
```
